# Historia de Usuario

## ID
[BE-04]

## Título

Solicitudes de inscripción (Usuario)

## Descripción
**Como** usuario

**Quiero** solicitar participación en un curso y consultar el estado de mi solicitud

**Para** inscribirme en las ofertas formativas de mi interés y realizar el seguimiento de mi admisión.

## Criterios de Aceptación

### Escenario 1: Creación de solicitud y validación de duplicados

```gherkin
Dado que soy un usuario autenticado
Cuando envío una petición POST a "/api/applications" indicando un ID de curso válido
Entonces el sistema crea la solicitud con estado inicial "pending"
Y si intento solicitar el mismo curso nuevamente, el sistema devuelve un error 409 Conflict
```

### Escenario 2: Consulta de solicitudes propias
```gherkin
Dado que he realizado varias solicitudes de inscripción
Cuando accedo al endpoint GET "/api/applications/me"
Entonces el sistema devuelve un listado de mis solicitudes con el nombre del curso, fecha y estado actual
Y no se deben mostrar solicitudes de otros usuarios.
```
### Escenario 3: Cancelación de solicitud propia
```gherkin
Dado que tengo una solicitud en estado "pending" o "accepted"
Cuando envío una petición DELETE a "/api/applications/:id"
Entonces el sistema cambia el estado de la solicitud a "cancelled"
Y si intento cancelar la solicitud de otro usuario, recibo un error 403 Forbidden
```


## Notas
* **Contexto de FastAPI:** Usar request.state para persistir la identidad del usuario durante el ciclo de vida de la petición.
* **Jerarquía:** superadmin hereda permisos de admin, pero admin no hereda permisos de superadmin.
* **Enums:** Se deben usar constantes para los estados (pending, accepted, rejected, cancelled) para evitar errores de escritura.
* **Contexto:** La inyección en request.state permite que los controladores accedan al usuario actual sin volver a decodificar el token.
* **Seguridad:** El error 403 debe ser genérico para evitar la enumeración de roles o rutas existentes.
