# Historia de Usuario

## ID
[BE-05]

## Título

Gestión de solicitudes (admin)

## Descripción
**Como** administrador

**Quiero** ver, aceptar, rechazar y eliminar solicitudes de participación por curso

**Para** controlar el cupo de los cursos y formalizar la inscripción de los alumnos.

## Criterios de Aceptación

### Escenario 1: Consulta de solicitudes por curso

```gherkin
Dado que soy un administrador autenticado
Cuando realizo una petición GET a "/api/courses/:id/applications"
Entonces el sistema devuelve la lista de todas las solicitudes de ese curso
Y la respuesta incluye el nombre y email de cada usuario solicitante
```

### Escenario 2: Cambio de estado de la candidatura
```gherkin
Dado que existe una solicitud en estado "pending"
Cuando el administrador envía una petición PATCH a "/api/applications/:id/status" con el nuevo estado ("accepted" o "rejected")
Entonces el sistema actualiza el registro en la base de datos
Y si el estado enviado no es válido, el sistema devuelve un código 422
```
### Escenario 3: Eliminación física de registros
```gherkin
Dado que soy un administrador autenticado
Cuando envío una petición DELETE a "/api/applications/:id"
Entonces el sistema elimina permanentemente la solicitud del registro
Y si un usuario con rol "user" intenta realizar esta acción, recibe un error 403 genérico
```

## Notas
* **Contexto de FastAPI:** Usar request.state para persistir la identidad del usuario durante el ciclo de vida de la petición.
* **Jerarquía:** El superadmin hereda permisos de admin, permitiéndole realizar estas gestiones de igual forma.
* **Enums:** Se deben usar constantes para validar los estados permitidos y evitar errores de escritura.
* **Contexto:** La inyección en request.state permite asegurar que solo roles autorizados ejecuten estas acciones sin decodificar el token repetidamente.
* **Seguridad:** Los intentos de cambio de estado con rol "user" deben resultar en un error 403 genérico ("Insufficient permissions").
