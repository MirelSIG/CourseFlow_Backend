# Historia de Usuario

## ID
[BE-01]

## Título
Registro e inicio de sesión de usuarios

## Descripción
**Como** usuario
**Quiero** registrarme y autenticarme 

**Para** acceder a la plataforma de forma segura.

## Criterios de Aceptación

### Escenario 1: RRegistro de un nuevo usuario (POST /api/auth/register)
```gherkin
Dado que un usuario envía su email, contraseña y rol (user/admin)
Cuando el email no existe en la base de datos
Entonces el sistema debe aplicar hash a la contraseña con bcrypt
Y crear el registro con el rol asignado
Y devolver una respuesta exitosa (201 Created)
```

### Escenario 2: Intento de registro con email duplicado
```gherkin
Dado que un usuario intenta registrarse con un email ya existente
Cuando envía la petición al endpoint de registro
Entonces el sistema debe denegar la creación
Y devolver un error estandarizado (409 Conflict)
```

### Escenario 3: Inicio de sesión exitoso (POST /api/auth/login)
```gherkin
Dado que el usuario envía credenciales válidas (email y password)
Cuando el sistema verifica la coincidencia en la BD
Entonces el sistema debe devolver un Access Token JWT
Y el payload del token debe contener {user_id, role, exp}
Y devolver una respuesta exitosa (200 OK)
```

### Escenario 4: Intento de login con contraseña incorrecta
```gherkin
Dado que el usuario envía credenciales inválidas
Cuando el sistema no puede verificar la contraseña
Entonces el sistema debe denegar el acceso
Y devolver un error estandarizado (401 Unauthorized)
```

### Escenario 5: Cierre de sesión (POST /api/auth/logout)
```gherkin
Dado que un usuario tiene un token activo
Cuando envía la petición al endpoint de logout
Entonces el sistema debe invalidar el token (añadiéndolo a una lista negra en tabla o Redis)
Y devolver una respuesta 200 OK
```

## Notas
* **Seguridad:** Uso obligatorio de bcrypt para el hashing de contraseñas.
* **Testing:** Se requiere cobertura con pytest para: registro duplicado, login incorrecto y uso de token expirado.
