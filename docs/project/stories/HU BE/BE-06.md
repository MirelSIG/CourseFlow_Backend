# Historia de Usuario

## ID
[BE-06]

## Título

Perfil del usuario autenticado

## Descripción
**Como** usuario

**Quiero**consultar y actualizar mis datos de perfil

**Para** mantener mi información personal al día dentro de la plataforma.

## Criterios de Aceptación

### Escenario 1: Consulta de información personal

```gherkin
Dado que soy un usuario autenticado con un token válido
Cuando realizo una petición GET a "/api/users/me"
Entonces el sistema devuelve mi id, name, email y role
Y el campo password_hash no debe estar presente en la respuesta JSON
```

### Escenario 2: Actualización parcial del perfil
```gherkin
Dado que deseo cambiar mi nombre o correo electrónico
Cuando envío una petición PATCH a "/api/users/me"
Entonces el sistema valida que, si el email cambia, no exista ya en la base de datos
Y devuelve los datos actualizados con un código 200
```
### Escenario 3: Intento de acceso no autorizado
```gherkin
Dado que no envío un token de autenticación
Cuando intento acceder a los endpoints de perfil
Entonces el sistema debe denegar el acceso inmediatamente
Y devolver un código de error 401 Unauthorized
```

## Notas
* **Contexto de Flask:** Usar flask.g para persistir la identidad del usuario durante el ciclo de vida de la petición.
* **Jerarquía:**superadmin hereda permisos de admin, pero admin no hereda permisos de superadmin.
* **Enums:** Se deben usar constantes para evitar errores de escritura (typos) en la lógica de comparación.
* **Contexto:** La inyección en flask.g permite que los controladores accedan al usuario actual sin volver a decodificar el token.
* **Seguridad:** El error 403 debe ser genérico para evitar la enumeración de roles o rutas existentes.
