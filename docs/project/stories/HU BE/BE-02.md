# Historia de Usuario

## ID
[BE-02]

## Título
Middleware de Autenticación y Autorización por Jerarquía

## Descripción
**Como** sistema
**Quiero** que cada ruta protegida valide el JWT y el rol antes de ejecutarse

**Para** asoportar una jerarquía completa de acceso (user < admin < superadmin) y garantizar la seguridad de los recursos.

## Criterios de Aceptación

### Escenario 1: Validación de identidad (@require_auth)

```gherkin
Dado que un cliente realiza una petición a una ruta protegida
Cuando envía el JWT en el header 'Authorization: Bearer <token>'
Entonces el decorador debe validar la firma, la expiración y la lista negra
Y si es válido, inyectar en 'flask.g.current_user' el objeto {id, role}
Y responder 401 si el token es inválido o está en lista negra
```

### Escenario 2: Autorización por jerarquía (@require_role)
```gherkin
Dado que una ruta está protegida con @require_role(["admin"])
Cuando un 'superadmin' intenta acceder
Entonces el sistema debe permitir el acceso implícito por jerarquía
Y si un 'user' intenta acceder, debe responder 403
```

### Escenario 3: Privacidad en errores de autorización
```gherkin
Dado que un usuario tiene permisos insuficientes para una ruta
Cuando el middleware deniega el acceso
Entonces el sistema debe devolver un error 403 genérico
Y el mensaje debe ser "Insufficient permissions" sin revelar requisitos de la ruta
```

### Token inválido o expirado
```gherkin
Dado un token que ha expirado o está en la lista negra (blacklist)
Cuando el middleware procesa la petición
Entonces el sistema debe detener la ejecución inmediatamente
Y devolver una respuesta 401 Unauthorized
```

## Notas
* **Contexto de Flask:** Usar flask.g para persistir la identidad del usuario durante el ciclo de vida de la petición.
* **Jerarquía:**superadmin hereda permisos de admin, pero admin no hereda permisos de superadmin.
* **Enums:** Se deben usar constantes para evitar errores de escritura (typos) en la lógica de comparación.
* **Contexto:** La inyección en flask.g permite que los controladores accedan al usuario actual sin volver a decodificar el token.
* **Seguridad:** El error 403 debe ser genérico para evitar la enumeración de roles o rutas existentes.
