# Historia de Usuario

## ID
[BE-07]

## Título

Gestión de usuarios admin (Superadmin)

## Descripción
**Como** superusuario

**Quiero** gestionar las cuentas de nivel administrativo y asegurar mi propio acceso mediante variables de entorno

**Para** administrar al personal de la plataforma con privilegios máximos sin depender de procesos manuales de base de datos.

## Criterios de Aceptación

### Escenario 1: Autoinstalación (Bootstrap) del Superadmin

```gherkin
Dado que el sistema se inicia por primera vez (o mediante un script de seed)
Cuando el proceso lee las variables "SUPERADMIN_EMAIL" y "SUPERADMIN_PASSWORD"
Entonces se crea automáticamente la cuenta con rol 'superadmin'
Y si la cuenta ya existe, el sistema omite la creación para evitar duplicados
```

### Escenario 2: Gestión de cuentas administrativas (POST /api/admin/users)
```gherkin
Dado que soy el superadmin autenticado
Cuando envío los datos de un nuevo administrador (name, email, password)
Entonces el sistema crea la cuenta con rol 'admin' y valida que el email sea único
Y si un usuario con rol 'admin' o 'user' intenta esta acción, recibe un error 403
```
### Escenario 3: Restricción de auto-eliminación
```gherkin
Dado que soy el superadmin autenticado
Cuando intento enviar una petición DELETE a mi propio ID de usuario
Entonces el sistema debe denegar la acción con un código 400 Bad Request
Y mostrar un mensaje indicando que el superadmin no puede eliminarse a sí mismo
```
### Escenario 4: Listado exclusivo de administradores (GET /api/admin/users)
```gherkin
Dado que el sistema tiene múltiples roles (user, admin, superadmin)
Cuando el superadmin solicita el listado de gestión
Entonces el sistema solo debe devolver los usuarios con rol 'admin'
Y omitir de la lista a los usuarios normales y al propio superadmin
```

## Notas

* **Contexto de FastAPI:** Usar request.state para persistir la identidad del usuario durante el ciclo de vida de la petición.
* **Jerarquía:** superadmin hereda permisos de admin, pero las rutas de esta HU son exclusivas para superadmin mediante el decorador específico.
* **Enums:** Se deben usar las constantes Role.SUPERADMIN y Role.ADMIN definidas en enums.py.
* **Contexto:** No existe endpoint público para la creación del primer superadmin; la seguridad depende totalmente del archivo .env.
* **Seguridad:** El error 403 debe ser genérico ("Insufficient permissions") para evitar la enumeración de la existencia de rutas de superusuario.
