# Historia de Usuario

## ID
[BE-03]

## Título

CRUD completo de cursos (Gestión Administrativa)

## Descripción
**Como** administrador

**Quiero** crear, editar, activar/desactivar y eliminar cursos desde la API

**Para** gestionar la oferta académica disponible en la plataforma de manera centralizada.

## Criterios de Aceptación

### Escenario 1: Creación de curso con validación de fechas (POST /api/courses)

```gherkin
Dado que soy un administrador autenticado
Cuando envío una petición POST a "/api/courses" con 'end_date' anterior a 'start_date'
Entonces el sistema debe rechazar la creación con un código 422
Y si las fechas son correctas, debe devolver un código 201 con el curso creado
```

### Escenario 2: Listado de cursos filtrado por rolo (GET /api/courses)
```gherkin
Dado que existen cursos activos e inactivos en la base de datos
Cuando un usuario con rol 'user' solicita el listado vía GET "/api/courses"
Entonces solo debe recibir los cursos donde 'is_active' sea verdadero
Y si el solicitante es 'admin', debe recibir la lista completa
```
### Escenario 3: Edición parcial y borrado lógico
```gherkin
Dado un curso existente
Cuando un administrador envía una petición PUT/PATCH para modificar solo la 'capacity'
Entonces el sistema debe actualizar solo ese campo y mantener el resto igual
Cuando un administrador envía una petición DELETE, el sistema debe marcar 'is_active=false'
```
### Escenario 4: Protección de rutas y seguridad
```gherkin
Dado que un usuario intenta acceder a POST, PUT o DELETE "/api/courses"
Cuando el usuario no tiene un token válido o no tiene rol 'admin'
Entonces el sistema debe devolver un error 401 (sin token) o 403 (rol insuficiente)

```

## Notas
- Se debe usar Flask-SQLAlchemy para el modelo de datos.
- Los decoradores @require_auth y @require_role("admin") son obligatorios para los métodos de escritura.
- El borrado será lógico por defecto para no romper la integridad de las inscripciones (applications) existentes.
