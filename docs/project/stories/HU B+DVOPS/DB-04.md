# Historia de Usuario

## ID
[DB-04]

## Título
Tests de integración con pytest

## Descripción
**Como** equipo de desarrollo

**Quiero** disponer de una suite de tests que valide endpoints reales contra una base de datos de prueba

**Para** asegurar que el flujo completo de la aplicación (backend y base de datos) funcione correctamente y evitar regresiones antes de cada entrega.

## Criterios de Aceptación

### Escenario 1: Configuración del entorno de pruebas (Fixtures)
```gherkin
Dado que el proyecto utiliza Pytest para la validación del código
Cuando se ejecuta la suite de pruebas
Entonces el archivo "conftest.py" debe proveer una instancia de la aplicación Flask conectada a una DB SQLite en memoria
Y debe existir una fixture "auth_headers" que registre un usuario y devuelva un token JWT válido para peticiones protegidas
```

### Escenario 2: Cobertura y volumen de pruebas
```gherkin
Dado que se requiere garantizar la estabilidad de la API
Cuando se ejecutan los tests de integración
Entonces se deben validar los flujos de: Registro, Login, CRUD de Cursos y Flujo de Solicitud completo
Y la suite debe contar con al menos 15 tests, alcanzando una cobertura mínima del 70% en rutas y modelos
```

### Escenario 3:Aislamiento y atomicidad de los tests
```gherkin
Dado que múltiples tests se ejecutan de forma secuencial
Cuando un test modifica datos en la base de datos de prueba
Entonces el sistema debe realizar un rollback o recrear las tablas al finalizar el test
Y ningún test debe depender del estado o de los datos creados por un test anterior
```

## Notas

- **Contexto de Flask:** Usar flask.g para persistir la identidad del usuario durante el ciclo de vida de la petición.
- **Jerarquía:** Los tests deben validar que la jerarquía de roles (User, Admin, Superadmin) se aplique correctamente en los endpoints.
- **Enums:** Las pruebas de integración deben verificar que los estados de las aplicaciones (pending, accepted, etc.) cambien según la lógica de negocio definida.
- **Contexto:** Se utilizará pytest-cov para generar reportes de cobertura directamente en la terminal o archivos HTML.
- **Seguridad:** El sistema debe validar que los errores de autorización devuelvan mensajes genéricos para no exponer información de la estructura interna.
