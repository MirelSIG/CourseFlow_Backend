## Título
Dockerización completa del proyecto

## Descripción
**Como** equipo de desarrollo

**Quiero** poder levantar todo el stack (backend, frontend y base de datos) con un solo comando

**Para** simplificar el despliegue, asegurar la paridad entre entornos y facilitar la incorporación de nuevos desarrolladores.

## Criterios de Aceptación

### Escenario 1: Construcción de la imagen del Backend
```gherkin
Dado que el proyecto utiliza Python 3.11
Cuando se ejecuta el proceso de build de Docker
Entonces se debe crear una imagen basada en "python:3.11-slim"
Y debe instalar todas las dependencias listadas en "requirements.txt" de forma aislada.
```

### Escenario 2: Orquestación con Docker Compose
```gherkin
Dado un entorno sin servicios previos instalados
Cuando se ejecuta el comando "docker compose up --build"
Entonces se deben levantar tres servicios: "backend" (FastAPI/Uvicorn), "db" (PostgreSQL 15) y "frontend" (Vite)
Y el backend debe esperar a que la base de datos esté lista mediante un "healthcheck".
```

### Escenario 3: Persistencia de datos y migraciones automáticas
```gherkin
Dado que los datos de la base de datos deben sobrevivir al reinicio de los contenedores
Cuando el servicio "db" se inicia, debe utilizar un volumen persistente
Y el "entrypoint" del backend debe ejecutar automáticamente "alembic
```

### Escenario 4: Poblamiento Inicial (Seed) y Documentación
```gherkin
Dado un entorno de base de datos recién migrado
Cuando se ejecuta el script "seed.py" (vía "python scripts/seed.py")
Entonces el sistema debe insertar 2 admins, 5 users, 3 cursos y 10 solicitudes de prueba
Y el diagrama ERD debe estar visible en el README para consulta del equipo
```

## Notas

- **Contexto de FastAPI**: Usar request.state para persistir la identidad del usuario durante el ciclo de vida de la petición.
- **Jerarquía**: El entorno dockerizado debe soportar la configuración de superadmin a través de las variables de entorno inyectadas en el docker-compose.yml.
- **Enums**: Las variables de entorno de la base de datos deben coincidir con las definidas en el archivo .env.
- **Contexto**: El uso de un entrypoint.sh es obligatorio para asegurar que las migraciones de base de datos se apliquen de forma secuencial y automática.
- **Seguridad**: El archivo .env con las credenciales de producción nunca debe ser incluido en la imagen de Docker (se debe inyectar en tiempo de ejecución).
