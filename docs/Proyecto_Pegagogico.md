## Descripción rápida
Realizar una aplicación web que contenga un frontend y un backend que solucione la expectativa de funcionalidad de una entidad en todo el recorrido fullstack. Que pueda consumir y gestionar una API propia con un CRUD completo.

Se adjudican proyectos reales para realizar en producción, con un stakeholder (cliente) que contraste con el equipo del proyecto la lista de requerimientos y la presentación de los logros obtenidos en las iteraciones de proyecto o sprints.

El modelo de proyecto que se adjudica a los grupos de trabajo por parte del formador, en este caso:

- Aplicación web fullstack orientada a la gestión de cursos y solicitudes de participación en entornos formativos y sociales como Somos F5. Trabajo que hasta el momento se realiza de forma manual.

## Contexto del proyecto
Hay que realizar una aplicación web completa que contenga las partes necesarias para ofrecer una funcionalidad propia, original y completa.

No se puede utilizar una IA (Inteligencia Artificial) como elemento que realice la aplicación. Aunque sí es posible usarla como herramienta de apoyo o ayuda, utilizando agentes integrados en el proyecto con parámetros de referencia SCRUM (contexto, historias de vida, tareas, etc.).

Desde un frontend realizado con una SPA (Single Page Application) que gestione una API de datos y un CRUD completo con un backend estructurado en un lenguaje que contenga la lógica de la aplicación con una base de datos SQL, que trabaje sobre el modelo relacional.

El stack a utilizar es el que se ha dado en el roadmap o itinerario del curso: FIGMA, HTML5, CCS3, JAVASCRIPT, VUE, PYTHON, FASTAPI y bases de datos SQL (MySQL, SqLite, PostgreSQL, MongoDB), utilizando las herramientas con las que hemos trabajado: VSC, JIRA, GIT, GITHUB, SASS, TAILWIND, BOOTSTRAP. El cliente puede variar algún elemento de esta lista que se entienda como requerimiento.

Hay que crear un CRUD completo que permita crear, modificar y/o borrar los datos que den funcionalidad al sistema del cual se conoce su finalidad y objetivo.

Dependiendo del sentido del proyecto, el acceso a la base de datos y la capacidad para llevar a cabo esta interacción será de los administradores (a través de una intranet), de los usuarios de la aplicación o de los visitantes.

Este acceso múltiple deberá ser autenticado y proporcionar permisos y privilegios de acceso a la BBDD según grupos de usuarios.

## Modalidades pedagógicas
### Secuencia pedagógica
Trabajo en grupos de 5 personas. Se configuran 3 grupos de proyecto. Cada grupo de trabajo tiene que generar una identidad de marca que identifique su autoría y funcionamiento, creando:

- Logo.
- Leitmotiv o frase fuerza.
- Nombre de la entidad que genera el proyecto.

Estos datos que aportan una identidad de grupo con un nombre o denominación específica y un logo estarán presentes en la página de proyecto y también en el README.

En cada grupo se designa por el formador la figura de Scrum Master y la persona que será Product Owner. También se identifica con claridad el Development Team. El equipo trabaja con metodología SCRUM y Agile, y atenderá todas las ceremonias y artefactos de la metodología: sprint planning, dailys, product backlog, sprint backlog, sprint retro, etc.

Los grupos son autónomos para la realización del proyecto.

Al final del proyecto cada equipo mostrará en un sprint review al stakeholder y al resto de los equipos de desarrollo las funcionalidades desarrolladas en el sprint con el producto mínimo viable. La demostración tendrá un tiempo mínimo de 15 minutos y máximo de una hora.

Con posterioridad cada grupo realizará un sprint retro en la que se evalúan los aspectos básicos del proceso. Se realiza frente a un panel utilizando pegatinas según el modelo Kanban.

### Herramientas
El framework de frontend será Vue 3 (Composition API). En algún proyecto, a requerimiento del cliente, se puede considerar otro framework, con la validación del formador.

El framework de backend a utilizar en todos los casos y sin posibilidad de cambio es FastAPI (Fast API, puede también ser considerado como opción) del lenguaje de programación Python, con todas sus librerías. Se puede utilizar un ORM como Alchemy, o utilizar directamente el lenguaje SQL y acreditar su manejo (criterio también consultado con diversas empresas).

Utilizar un modelo de bases de datos relacionales en lenguaje SQL: MySQL-MariaDB, PostgreSQL, SqLite, para usar funcionalidades medias y avanzadas. Es posible realizar un soporte de datos en un modelo No-SQL, pero deberá estar justificado y validado. Generar en cada caso el proceso completo de creación de un modelo de BBDD y acreditarlo en el documento de presentación.

## Criterios de rendimiento
- El material propio de desarrollo, como el código fuente, ha sido entregado en modo y plazo correcto.
- Se han utilizado aplicaciones de control de versiones tal y como atestigua el repositorio remoto facilitado al efecto. Utilización de Git y GitHub.

### Requerimientos técnicos
- Análisis de la interfaz bajo los criterios UI/UX. User Experience.
- SEO: elementos de etiquetas y archivos necesarios para la optimización del posicionamiento de la página por los buscadores. Title Tag, Meta description (adaptado a todas las páginas), robots.txt, sitemap.xml.
- La herramienta de trabajo colaborativo a utilizar por los grupos será Jira.
- Plataforma multilingüe. Crear un site que contenga dos idiomas fijos (castellano, euskara o inglés), un idioma preparado en script pero comentado (euskara) y posibilidad en código escalable a otro idioma más por lo menos. La web bilingüe tendrá páginas de los elementos constitutivos de la misma. Se adaptará a los criterios de cliente.
- Se trabaja con Vue 3, con la configuración de composition. Se recomienda utilizar Vite. Se podrán utilizar las modalidades acreditadas como composition api.
- Se crearán diversas views o pages que den un carácter dinámico y completo al website a través de diversos contenidos accesibles a los usuarios.
- Se utilizará la modularización creando componentes para poder albergar los datos de visualización en un módulo que se pueda replicar en bucle (una lista desordenada, o unos section dentro de un article).
- Se utilizarán las directivas que permiten dar comportamientos integrados al desarrollo, cada cual en su modus operandi: v-bind para renderizar datos, v-show para mostrar y ocultar, v-if para condicionales o v-for para las iteraciones.
- Se utilizarán archivos router para marcar las rutas, los prop para información entre los módulos parent y child y los eventos emit para hacer lo correspondiente en sentido inverso.
- O en su caso utilizar plugins o librerías para gestionar estados globales como Pinia.
- Se utilizará la reactividad de datos primarios y de objetos (librerías ref y reactive).
- El proyecto será totalmente responsive con al menos dos breakpoints (diseño para mobile, tablet y desktop). Se considera un valor añadido realizarlo en First Mobile.
- Bocetado completo del website en Figma para los tres ámbitos del dispositivo. Con versión en blanco y negro y en color.
- Dentro de la utilización de Git y GitHub, es de interés que se utilicen ramas (Branch) en la realización del proyecto, creando las necesarias para garantizar la autonomía y la capacidad de tener líneas de trabajo de respaldo (con rama de producción, de desarrollo y de trabajo individual). Acreditarlo de forma gráfica en el documento de presentación. (Git Kraken).
- El README será explicativo de todas las partes del proyecto. Contará con imágenes en su diseño interno.
- Los scripts de código desarrollados funcionan de forma eficiente, sin bugs y sin problemas de renderización. Anteriormente a su presentación el código ha sido probado y testado. El código está formateado y comentado adecuadamente.
- El código tanto en la designación de elementos arbitrarios (nombres de variables, objetos, funciones, etc.) y de comentarios (de código) será realizado enteramente en inglés, y cuidará que sea explicativo y semántico.
- El código deberá estar explicado y comentado para facilitar futuras labores de mantenimiento por parte del equipo de creación o de un futuro equipo que realice esa labor.
- Realizar scripts de testing en frontend utilizando una librería como Jest, Jasmine o Vue Test o Cypress.
- Es necesario utilizar técnicas de testing en backend con pytest, en varios de los scripts desarrollados en los archivos de la aplicación, aunque no sea la totalidad de los mismos.
- El proyecto utilizará Docker para poder crear un sistema integrado y flexible para facilitar su despliegue en un servidor remoto. Crear la dockerización del proyecto en base a los estándares de otros proyectos BBDD.
- Se tienen en cuenta los principios SOLID, en la creación de los scripts del código.
- La arquitectura interna de los distintos directorios de carpetas y archivos reflejará los principios de modularización y segmentación de funciones, para mejorar la mantenibilidad y la responsabilidad específica. Buenas prácticas en la realización de código.
- El desarrollo de backend deberá proporcionar la respuesta adecuada a las necesidades de gestión y administración de la web. Habrá que consultar con el stakeholder qué funcionalidades deberán hacer peticiones que cubran las expectativas necesarias.
- El sistema deberá contar con un sistema de autenticación para preservar la seguridad en el acceso. Mediante JSON Web Tokens, id de sesión o los elementos que aseguren este ámbito de desarrollo.
- La demostración es acertada, de forma concisa y clara. Se ha respondido a la demanda de información adicional de forma adecuada.
- La actitud y las habilidades de comunicación en la exposición de la tarea realizada son propias del estándar profesional en el ámbito del desarrollo en programación y a los usos habituales en contactos profesionales.
- La presentación correrá a cargo de la totalidad del equipo y contará con el apoyo de todos sus miembros en todo momento. La presentación se realiza como un equipo de trabajo cohesionado, minimizando las discrepancias internas y mostrando apoyo mutuo.
- Se utilizará la modularización y, en los casos en los cuales tenga sentido y funcionalidad, se utilizará POO, con clases que instancien objetos con atributos y acciones, que se puedan invocar o importar desde otra parte del código.
- El diseño de la API tendrá como referencia los criterios de API REST, para asegurar la adecuada interpretación de los endpoints por parte de la comunidad de programadores, respetando el estándar.
